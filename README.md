
# Business Process Service

Backend-сервис для управления жизненным циклом бизнес-процессов (дел).  
Система реализует контролируемую статусную модель, хранит историю изменений, управляет участниками процессов и документами, а также обеспечивает прозрачную бизнес-логику переходов между состояниями дела.

Основная задача сервиса — предоставить API для управления делами, контролируя допустимые переходы статусов, фиксируя историю изменений и обеспечивая целостность бизнес-процессов.

Проект реализован на Java с использованием Spring Boot и предназначен для работы в архитектуре REST API.

---

# Назначение системы

Сервис управляет бизнес-процессами, представленными в виде дел (Cases).  
Каждое дело проходит определённый жизненный цикл — от создания до архивирования.

Система обеспечивает:

- управление делами
- управление участниками процессов
- контроль ролей участников
- хранение документов
- контроль переходов между статусами
- хранение истории изменений статусов
- автоматические действия при смене статуса

Такая архитектура позволяет реализовать прозрачную и управляемую модель бизнес-процессов.

# Архитектура приложения

Приложение построено по принципам слоистой архитектуры.  
Каждый слой отвечает за свою зону ответственности и не содержит логики других слоев.

Структура взаимодействия слоев:

```

Controller → Service → Repository → Database

```

## Controller слой

Контроллеры принимают HTTP-запросы от клиента и преобразуют их во внутренние вызовы сервисов.  
Контроллеры не содержат бизнес-логики и выполняют только следующие задачи:

- обработка HTTP запросов
- валидация входных DTO
- вызов сервисного слоя
- формирование HTTP ответа

Такая структура делает код контроллеров простым и читаемым.

---

## Service слой

Сервисный слой содержит основную бизнес-логику приложения.

В сервисах реализованы:

- правила переходов статусов
- бизнес-валидация
- транзакционные операции
- автоматические действия при смене статуса

Именно этот слой реализует поведение системы.

---

## Repository слой

Repository слой обеспечивает доступ к базе данных.

Используется Spring Data JPA, что позволяет автоматически генерировать большинство операций работы с БД.

Репозитории отвечают за:

- поиск данных
- сохранение сущностей
- удаление данных
- фильтрацию и пагинацию

---

## DTO слой

DTO (Data Transfer Objects) используются для передачи данных между слоями.

Это позволяет:

- отделить внутреннюю модель данных от API
- контролировать формат данных
- избежать утечек внутренней структуры сущностей

---

# Доменная модель

Система оперирует несколькими основными сущностями.

---

## Case (дело)

Case является центральной сущностью системы.

Каждое дело представляет собой отдельный бизнес-процесс.

Основные поля:

| Поле | Описание |
|---|---|
id | уникальный идентификатор |
caseNumber | номер дела |
procedureType | тип процедуры |
status | текущий статус |
startDate | дата начала |
endDate | дата завершения |

Связанные сущности:

- участники дела
- документы
- история изменений статусов

---

## Party (участник дела)

Party представляет участника бизнес-процесса.

Участником может быть:

- физическое лицо
- юридическое лицо

Основные характеристики участника:

- тип участника
- отображаемое имя
- роль в процессе

---

## PartyRole

PartyRole определяет роль участника в рамках дела.

Примеры ролей:

```

DEBTOR
CREDITOR
APPLICANT
RESPONDENT

```

Роли используются для определения функции участника в процессе.

---

## ProcedureType

ProcedureType определяет тип бизнес-процедуры.

Например:

```

BANKRUPTCY
COURT_PROCEEDING
DOCUMENT_VERIFICATION

```

Тип процедуры влияет на бизнес-логику и возможные действия в процессе.

---

## Document

Document представляет документ, связанный с делом.

Документы используются для подтверждения этапов процесса.  
Например, завершение дела невозможно без наличия документов.

---

## StatusHistory

StatusHistory хранит историю изменения статусов дела.

Каждое изменение статуса фиксируется как отдельная запись.

Основные поля:

| Поле | Описание |
|---|---|
caseId | идентификатор дела |
previousStatus | предыдущий статус |
newStatus | новый статус |
changedAt | дата изменения |
initiatedBy | инициатор изменения |

Это обеспечивает полный аудит всех изменений процесса.

---

# Статусная модель процесса

Каждое дело проходит последовательный жизненный цикл.

В системе используются следующие статусы:

```

CREATED
IN_PROGRESS
PROCEDURE_RUNNING
COMPLETED
ARCHIVED

```

---

# Разрешённые переходы статусов

Система реализует строгие правила переходов между статусами.

Допустимая последовательность:

```

CREATED → IN_PROGRESS
IN_PROGRESS → PROCEDURE_RUNNING
PROCEDURE_RUNNING → COMPLETED
COMPLETED → ARCHIVED

```

Любой другой переход считается запрещённым и блокируется системой.

---

# Бизнес-ограничения переходов

Некоторые переходы требуют выполнения дополнительных условий.

### Переход в PROCEDURE_RUNNING

Для запуска процедуры необходимо наличие хотя бы одного участника дела.

Если участники отсутствуют, переход блокируется.

---

### Переход в COMPLETED

Для завершения процесса необходимо наличие хотя бы одного документа.

Это гарантирует, что процесс завершён корректно.

---

# Автоматические действия при смене статуса

При изменении статуса система выполняет дополнительные действия.

## Запись истории изменений

Каждый переход статуса автоматически записывается в таблицу истории.

---

## Пересчёт метрик

В системе предусмотрен механизм пересчёта связанных показателей.

На текущем этапе реализован stub (заглушка), который может быть расширен в будущем.

---

## Отправка уведомлений

После изменения статуса может быть отправлено уведомление.

На текущем этапе уведомление реализовано как stub и выводится в лог.

В будущем возможна интеграция с:

- email уведомлениями
- message broker
- системой событий

---

# API возможности

Сервис предоставляет REST API для управления системой.

Доступны следующие операции:

- управление делами
- управление участниками
- управление документами
- управление ролями
- управление типами процедур

Дополнительно реализованы:

- фильтрация дел
- поиск по номеру дела
- пагинация
- получение истории статусов

---

# Пагинация

Для получения списка дел используется стандартная пагинация Spring.

Пример запроса:

```

GET /api/cases?page=0&size=10

```

---

# Поиск дел

Дела можно искать по номеру.

```

GET /api/cases?caseNumber=CASE-001

```

---

# Фильтрация по статусу

```

GET /api/cases?status=CREATED

```

---

# Получение истории статусов

История изменений доступна через API.

```

GET /api/cases/{id}/history

```

---

# Примеры API запросов

## Создание дела

```

POST /api/cases

````

Пример тела запроса:

```json
{
  "caseNumber": "CASE-001",
  "procedureType": "BANKRUPTCY"
}
````

---

## Получение дела

```
GET /api/cases/{id}
```

---

## Обновление дела

```
PATCH /api/cases/{id}
```

---

## Удаление дела

```
DELETE /api/cases/{id}
```

Удаление допускается только если:

* статус дела CREATED
* у дела нет участников
* у дела нет документов

---

## Изменение статуса

```
POST /api/cases/{id}/status
```

Пример запроса:

```json
{
  "newStatus": "IN_PROGRESS",
  "initiatedBy": "system"
}
```

---

# Обработка ошибок

В системе реализован централизованный обработчик ошибок.

Примеры ошибок:

```
404 — Case not found
400 — Validation error
403 — Forbidden status transition
```

Это обеспечивает единый формат ошибок API.

---

# Транзакционность

Критические операции выполняются внутри транзакций.

Используется аннотация:

```
@Transactional
```

Это гарантирует:

* целостность данных
* атомарность операций
* корректную работу бизнес-процессов

# Запуск проекта

## Требования

Для запуска необходимо установить:

* Java 17
* PostgreSQL
* Maven

---

## Настройка базы данных

Создать базу данных:

```
business_process_db
```

---

## Конфигурация приложения

Пример `application.yml`:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/business_process_db
    username: postgres
    password: postgres

  jpa:
    hibernate:
      ddl-auto: update
    show-sql: true
```

```
mvn clean install
```

## Запуск

```
mvn spring-boot:run
```

Или запуск через IDE.
